# Modify Record Skill

在每次对话中，当代码修改被用户同意执行后，自动将修改记录保存到项目文件夹内的 `{项目名}_Modify/` 目录中。

## 为什么需要这个技能

代码修改往往散落在多次对话中，时间一长就难以回溯"某天改了什么、为什么改"。这个技能通过自动记录每次修改的上下文和 diff，让你随时能查看项目的变更历史，方便后期对比和审计。

## 目录与文件结构

在项目根目录下创建修改记录目录：

```
{项目根目录}/
├── {项目名}_Modify/
│   ├── {项目名}_Modify_2026-06.md          ← 月度索引
│   ├── {项目名}_Modify_2026-06-28.md
│   ├── {项目名}_Modify_2026-06-29.md
│   └── ...
└── .gitignore   ← 自动追加忽略规则（如果需要）
```

- **目录名**：`{项目名}_Modify/`，其中 `{项目名}` 取项目根目录的文件夹名
- **日期文件名**：`{项目名}_Modify_{yyyy-mm-dd}.md`，按天分文件
- **月度索引文件名**：`{项目名}_Modify_{yyyy-mm}.md`，每月一个索引
- **同一天的多次修改追加到同一个文件中**，不覆盖

## 忽略规则

修改记录目录本身不应被 git 追踪。在项目根目录的 `.gitignore` 中追加：

```
*_Modify/
```

如果 `.gitignore` 已存在且包含此规则，则跳过。如果不存在 `.gitignore`，则创建它。

## 非项目文件过滤

以下文件/目录属于自动生成、依赖管理或构建产物，对其修改**不记录**：

- **依赖目录**：`node_modules/`、`vendor/`、`venv/`、`.venv/`、`__pycache__/`
- **构建产物**：`dist/`、`build/`、`out/`、`.next/`、`coverage/`
- **锁文件**：`package-lock.json`、`yarn.lock`、`pnpm-lock.yaml`、`Cargo.lock`、`poetry.lock`
- **IDE 配置**：`.vscode/`、`.idea/`、`.DS_Store`
- **日志/临时文件**：`*.log`、`*.tmp`、`*.swp`

判断逻辑：如果被修改的文件路径**以这些目录/文件名开头或匹配这些模式**，则跳过记录。

## 何时记录

仅当以下条件**全部满足**时才触发记录：

1. **发生了实际的文件修改**：使用了 Edit 或 Write 工具写入了文件内容，或者通过 Bash 执行了会修改文件系统的命令（rm、mv、sed -i、echo > file 等）
2. **用户已同意执行**：修改已经实际生效，而非被拒绝、取消或跳过
3. **目标文件不在记录目录内**：被修改的文件路径不匹配 `*_Modify/` 模式（避免自引用循环）

### 触发时机

在修改操作**执行完毕后**立即触发记录，不是在用户提出需求时，也不是在讨论方案时。

### 明确不触发的情况

以下场景**不触发**此技能：

- 读取代码：使用 Read、cat、head、tail、grep 等只读操作
- 分析代码：讨论代码逻辑、review 代码、解释代码功能
- 讨论方案：用户提出需求但尚未执行修改，或正在讨论实现方案
- 查看文件：ls、find、tree 等查看文件结构的操作
- 用户拒绝修改：修改被用户取消或跳过
- 用户明确跳过：用户说"不用记录"或"跳过记录"
- 修改记录目录本身：目标文件在 `*_Modify/` 内

## 用户控制

用户可以通过以下指令控制记录行为：

| 指令 | 效果 |
|------|------|
| "暂停记录" / "暂停修改记录" | 当前对话中停止记录，直到用户说"恢复记录" |
| "恢复记录" / "继续记录" | 恢复自动记录 |
| "这次跳过" / "不用记录" | 仅跳过当前这一次修改的记录，下次修改照常记录 |
| "强制记录" / "这次也要记录" | 即使文件在忽略列表中也强制记录（如修改了 `package-lock.json` 的版本） |

这些指令在对话中自然语言即可，不需要特殊语法。

## 修改类型标签

每次记录都需要标注修改类型，方便后期按类型筛选和汇总。使用以下标签：

| 标签 | 含义 |
|------|------|
| `功能新增` | 添加新功能、新模块、新接口 |
| `bug修复` | 修复已知的 bug 或错误 |
| `重构` | 不改变功能的代码结构调整、优化 |
| `配置变更` | 修改配置文件、环境变量、构建脚本等 |
| `文件删除` | 删除文件或目录 |
| `文件重命名` | 重命名或移动文件 |
| `文档修改` | 修改 README、注释、文档等 |
| `依赖变更` | 添加、升级、移除依赖包 |

如果一次修改涉及多种类型，以主要类型为准，或用 `+` 连接（如 `功能新增+重构`）。

### 类型判断规则

根据修改内容自动判断类型，优先级从高到低：

1. **文件删除**：使用了 `rm`/`rmdir`，或删除了整个文件 → `文件删除`
2. **文件重命名**：使用了 `mv`，或重命名了文件 → `文件重命名`
3. **依赖变更**：修改了 `package.json`、`requirements.txt`、`Cargo.toml` 等依赖配置文件的依赖部分 → `依赖变更`
4. **配置变更**：修改了 `.env`、`tsconfig.json`、`webpack.config.*`、`Dockerfile`、`docker-compose.*`、CI 配置（`.github/workflows/`）等 → `配置变更`
5. **文档修改**：修改了 `README`、`CHANGELOG`、`*.md`（非代码注释）、JSDoc 注释批量添加 → `文档修改`
6. **bug修复**：用户需求中包含"修复"、"fix"、"bug"、"报错"、"error"等关键词 → `bug修复`
7. **功能新增**：新建了文件，或在已有文件中添加了新的函数/类/模块 → `功能新增`
8. **重构**：不改变外部行为的代码调整（重命名、移动、提取函数、简化逻辑等）→ `重构`

如果无法明确判断，默认使用 `重构`。

## 记录格式

### 格式设计原则

- **信息密度高**：一行元数据（时间 + 类型）+ 用户原话引用块，快速定位
- **代码用 diff 对比**：用 `diff` 代码块展示改动，`-` 表示删除、`+` 表示新增，一眼看出改了什么
- **视觉层次清晰**：用 emoji 图标区分不同信息块（💬 原话、📝 需求、📄 文件、🗑️ 删除、📎 重命名）
- **分隔线明确**：每条记录之间用 `---` 分隔，视觉上一目了然

### 修改记录（编辑/写入）

```markdown
---

### 🕐 yyyy-mm-dd HH:MM:SS | `修改类型`

> 💬 "{用户在对话中的原始表述}"

📝 **需求**：{对用户需求的简要概括}

📄 `{文件的相对路径}`

\`\`\`diff
  {上下文：未修改的行，前面加空格}
- {删除的行，前面加减号}
+ {新增的行，前面加加号}
  {上下文：未修改的行，前面加空格}
\`\`\`
```

### diff 格式规范

使用 unified diff 风格，代码块语言标注为 `diff`：

- **删除的行**：前面加 `- `（减号+空格）
- **新增的行**：前面加 `+ `（加号+空格）
- **未修改的上下文行**：前面加 ` `（一个空格），上下文保留 2-3 行即可，不需要展示整个文件
- 如果修改涉及多个不连续的区域，每个区域之间用空行分隔，或用 `@@ ... @@` 标记行号范围

### diff 构建要点

构建 diff 时容易犯的错误是把改动行混入上下文。记住这个原则：**每一行改动都必须有 `-` 或 `+` 前缀，绝不能作为普通上下文行出现**。

以 Edit 工具为例：`old_string` 是被替换的内容，`new_string` 是替换后的内容。

- `old_string` 中的**每一行**都加 `- ` 前缀
- `new_string` 中的**每一行**都加 `+ ` 前缀
- 改动区域的前后各保留 2-3 行未修改的代码作为上下文（加 ` ` 前缀）
- 如果上下文行和改动行相同（比如函数签名没变），上下文行用 ` ` 前缀，改动行用 `-`/`+` 前缀，不要混淆

**正确示例**（把 `UserService` 改名为 `AccountService`）：
```diff
- export class UserService {
+ export class AccountService {
    private baseUrl = '/api/users';
    async getUser(id: string) {
```

**错误示例**（改动行被当成上下文，看不到实际改了什么）：
```diff
  export class AccountService {
    private baseUrl = '/api/users';
    async getUser(id: string) {
```
上面的错误在于 `- export class UserService` 这行丢失了，读者无法知道原来叫什么。

### 删除记录

```markdown
---

### 🕐 yyyy-mm-dd HH:MM:SS | `文件删除`

> 💬 "{用户在对话中的原始表述}"

📝 **需求**：{简要概括}

🗑️ `{文件的相对路径}`

**删除原因**：{为什么删除，或记录删除前文件的大致用途}
```

### 重命名/移动记录

```markdown
---

### 🕐 yyyy-mm-dd HH:MM:SS | `文件重命名`

> 💬 "{用户在对话中的原始表述}"

📝 **需求**：{简要概括}

📎 `{原相对路径}` → `{新相对路径}`
```

### 多文件修改

如果一次用户需求涉及多个文件的修改，将所有修改合并到**同一条记录**中，每个文件用 `📄` 图标引导。

```markdown
---

### 🕐 yyyy-mm-dd HH:MM:SS | `修改类型`

> 💬 "{用户在对话中的原始表述}"

📝 **需求**：{统一的需求描述}

📄 `{文件相对路径1}`

\`\`\`diff
  {上下文}
- {删除的行}
+ {新增的行}
  {上下文}
\`\`\`

📄 `{文件相对路径2}`

\`\`\`diff
  {上下文}
- {删除的行}
+ {新增的行}
  {上下文}
\`\`\`
```

### 获取修改前后的代码

- **使用 Edit 工具时**：`old_string` 就是被替换的内容，`new_string` 就是替换后的内容。将 `old_string` 中的行标记为 `-`，`new_string` 中的行标记为 `+`，上下文保留 2-3 行。
- **使用 Write 工具时**：如果是在已有文件上覆盖写入，需要先读取原文件内容，与 Write 的新内容做 diff，只记录有差异的部分。如果是新建文件，整个文件内容都标记为 `+`。
- **使用 Bash 命令时**：
  - `rm` / `rmdir` → 记录为删除操作，格式同上
  - `mv` / `rename` → 记录为重命名操作，格式同上
  - `sed -i` / `echo > file` / `cat > file` → 尽量从命令推断修改内容，记录命令和受影响的文件
  - 批量操作（如 `find ... -exec`）→ 记录命令本身和涉及的目录/文件模式
  - 无法推断具体内容时 → 记录执行的完整命令和受影响的文件路径列表

### 路径规范

所有文件路径使用**相对于项目根目录的路径**，不使用绝对路径。这样记录在不同机器上都能正确对应文件位置。

## 同一文件多次修改的处理

如果同一个文件在同一天被修改了多次（可能在同一次对话中，也可能在不同对话中），每条修改作为**独立的记录**追加到当天的文件中。每条记录都有自己的时间戳和用户需求描述，天然形成了该文件的修改时间线，无需额外关联。

## 月度索引

每个月自动生成一个索引文件 `{项目名}_Modify_{yyyy-mm}.md`，记录当月所有修改的摘要。在每次写入日记录的同时更新月度索引。

### 月度索引格式

```markdown
# 📅 {项目名} 修改记录 - {yyyy年mm月} 索引

| # | 日期 | 时间 | 类型 | 用户需求 | 涉及文件 |
|---|------|------|------|----------|----------|
| 1 | 06-28 | 14:32 | `bug修复` | 修复登录页面的表单验证 bug | `src/components/LoginForm.tsx` |
| 2 | 06-28 | 16:45 | `功能新增` | 添加用户头像上传功能 | `src/api/upload.ts`, `src/components/Avatar.vue` |
| 3 | 06-29 | 10:15 | `重构` | 重构数据库连接模块 | `src/db/connection.ts` |

---

### 📊 本月统计

| 类型 | 次数 |
|------|------|
| 功能新增 | 1 |
| bug修复 | 1 |
| 重构 | 1 |
| **合计** | **3** |

- 涉及文件总数：4 个
```

### 月度索引的更新逻辑

每次往日记录文件追加新记录时，同步将该条记录的摘要行追加到月度索引的表格中，并更新底部的统计数据。如果月度索引文件不存在则创建。

## 执行步骤

当检测到代码修改已被用户同意执行后，按以下步骤操作：

1. **确定项目名**：取当前工作目录（项目根目录）的文件夹名
2. **创建目录**（如不存在）：`{项目根目录}/{项目名}_Modify/`
3. **处理 .gitignore**：确保 `*_Modify/` 规则存在
4. **确定日期文件**：`{项目名}_Modify_{yyyy-mm-dd}.md`
5. **确定月度索引文件**：`{项目名}_Modify_{yyyy-mm}.md`
6. **获取当前时间**：格式为 `yyyy-mm-dd HH:MM:SS`
7. **判断修改类型**：根据操作内容确定类型标签
8. **捕获用户原话**：从当前对话中提取用户的原始表述（而非摘要）
9. **概括用户需求**：对用户需求做简要概括
10. **获取修改内容**：根据使用的工具类型和操作类型，获取修改前后的代码或操作详情
11. **追加日记录**：将格式化后的记录追加到当天的 md 文件末尾
12. **更新月度索引**：将摘要行追加到月度索引表格，更新统计数据

### 新建日记录文件时的文件头

如果是当天第一次记录（文件刚创建），在文件开头添加：

```markdown
# 📅 {项目名} 修改记录 - {yyyy-mm-dd}

---
```

```markdown
📝 **需求**：修复登录页面邮箱格式校验过于宽松的 bug

📄 `src/components/LoginForm.tsx`

\`\`\`diff
  const validateEmail = (email) => {
-   return email.includes('@');
+   const re = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
+   return re.test(email);
  };
\`\`\`
```

### 示例 2：文件删除

```markdown
---

### 🕐 2026-06-28 16:10:33 | `文件删除`

📝 **需求**：删除已废弃的 helper 文件
```

```markdown
📝 **需求**：将用户相关组件归类到 user 目录，重命名 UserService

📎 `src/components/UserCard.vue` → `src/components/user/UserCard.vue`

📄 `src/services/UserService.ts`

+ export class AccountService {
    async getUser(id: string) { ... }

\`\`\`

📄 `src/components/UserProfile.tsx`

\`\`\`diff

\`\`\`
```

## 注意事项

- 时间戳使用**本地时间**，格式严格为 `yyyy-mm-dd HH:MM:SS`
- 文件路径一律使用**相对于项目根目录的路径**，不使用绝对路径
- 不要记录对 `*_Modify/` 目录本身的修改，这是为了避免无限递归
- 如果用户明确说"这次不用记录"或"跳过记录"，则不记录本次修改
- 修改记录是**追加写入**，永远不要覆盖或删除已有的记录
- 用户原话应尽量完整保留，不要过度概括；用户需求可以用一句话概括
- 代码块统一使用 `diff` 格式，用 `-` 和 `+` 标记增删行，上下文保留 2-3 行即可，不需要展示整个文件
- **emoji 图标必须使用**：🕐 时间、💬 用户原话、📝 需求、📄 文件、🗑️ 删除、📎 重命名，不要用文字替代（如不要写 `**文件删除**` 代替 `🗑️`）
- **分隔线只在记录之间使用**：`---` 只出现在两条修改记录之间，不要在同一条记录内部添加多余的 `---`

## 大文件 diff 截断

当修改的 diff 超过 50 行时，只保留**关键改动部分**，省略中间未修改的上下文。截断方式：

```diff
  // ... 前面的代码省略 (N 行) ...
  function changedPart() {
-   old code
+   new code
  }
  // ... 后面的代码省略 (M 行) ...
```

用 `// ... 省略 (N 行) ...` 注释标注省略的行数，让读者知道还有更多内容。

## 记录完成确认

每次记录写入后，给用户一行简短的确认消息（不要输出整个记录内容）：

```
✅ 已记录：{修改类型} | {涉及文件数}个文件 | {日期文件路径}
```

示例：
```
✅ 已记录：bug修复 | 1个文件 | test-project_Modify/test-project_Modify_2026-06-28.md
✅ 已记录：文件重命名+重构 | 3个文件 | web-app_Modify/web-app_Modify_2026-06-28.md
```

这条确认消息只在终端显示，不写入记录文件。
